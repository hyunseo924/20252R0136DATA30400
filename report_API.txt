### **[버전 1] 국문 정리 (내용 이해 및 초안용)**

#### **3.1. LLM 기반 데이터 생성 및 기술적 최적화**
본 프로젝트는 레이블이 없는 데이터셋의 한계를 극복하기 위해, Google Gemini API를 활용하여 고품질의 'Golden Validation Set'과 초기 학습용 'Seed Data'를 생성하였다. 이 과정에서 발생한 기술적 난관과 해결 방안은 다음과 같다.

**1) 네트워크 프로토콜 호환성 문제 (gRPC Blocking)
* 문제점: 초기 API 호출 시 에러 메시지 없이 프로그램이 무한 대기(Hang)하는 현상이 발생함.
* 원인: Google Generative AI SDK의 기본 통신 프로토콜인 gRPC가 실행 환경(Colab/Local Network)의 방화벽이나 포트 정책에 의해 차단됨을 확인.
* 해결: API 설정을 `transport='rest'`로 변경하여, 범용적인 HTTP/REST 프로토콜을 통해 안정적인 통신 채널을 확보함.

**2) API 속도 제한(Rate Limiting) 및 429 에러 관리
* 문제점: 일일 할당량(RPD)은 충분했으나, 반복문 실행 시 `429 Resource Exhausted` 에러가 빈번히 발생함.
* 원인: 실험적 모델인 `gemini-2.0-flash-exp`의 엄격한 분당 요청 수 제한(10 RPM, 6초당 1회)을 초과함. 단순 `limit: 0` 에러를 일일 한도 초과로 오인할 소지가 있었음.
* 해결:
    * 모델을 안정적인 `gemini-1.5-flash` (15 RPM)로 변경하거나, `time.sleep()`을 이용한 명시적 지연(Throttling) 로직을 구현함.
    * 단순 대기가 아닌 `Exponential Backoff` 알고리즘을 적용한 재시도(Retry) 래퍼 함수를 구현하여 API 호출의 안정성을 보장함.

**3) Kernel 재시작으로 환경 초기화 
* 문제점: 이전에 예시 코드로 성공한 사례가 있었으나 충분한 시간이 흐른 후 적절한 time.sleep()을 주었는데도 실행이 되지 않음. API 사용량을 직접 확인하여 요청 자체가 가지 않았음을 확인. 
* 원인: 이전에 존재했던 무한로딩(Hang)이 끝나지 않고 걸려있음. 
* 해결: 
    * kernel을 재시작하여 완전히 메모리를 비움. 
    * transport를 변경하지 않아도 정상 작동.
**4) 제한된 자원 내에서의 데이터 효율화 (Resource Management)
* 제약: 프로젝트 규정상 1,000회 미만의 API 호출만 허용됨.
* 전략: 전체 데이터 레이블링 대신, 학습의 나침반 역할을 할 **Validation Set (200개)**과 초기 학습의 닻(Anchor)이 될 **Seed Data (약 400~600개)** 생성에 집중함.
* 안전장치: 커널 중단(OOM 등) 시 API 호출 비용 손실을 방지하기 위해, 데이터 생성 즉시 디스크에 직렬화(Pickle/JSON)하여 저장하는 캐싱(Caching) 파이프라인을 구축함.

#### **3.1.1. High-Quality Label Generation via LLM**
To mitigate the label noise inherent in weak supervision, we generated a high-quality "Golden Validation Set" and "Seed Training Set" using the Google Gemini API. We addressed several technical challenges during this process:

**1. Protocol Adaptation (gRPC vs. REST):**
We initially encountered indefinite hanging issues during API calls. Investigation revealed that the default **gRPC** protocol used by the SDK was blocked by the network environment. We resolved this by explicitly configuring the client to use **REST (`transport='rest'`)**, ensuring stable connectivity.

**2. Rate Limit Handling:**
Despite sufficient daily quotas, we faced frequent `429 Resource Exhausted` errors due to the strict **Requests Per Minute (RPM)** limits of the free-tier models (e.g., 10 RPM for `gemini-2.0-flash-exp`). To address this, we implemented a robust **throttling mechanism** using `time.sleep()` combined with **Exponential Backoff** retry logic. This ensured compliance with API rate limits while maximizing throughput.

**3. Resource Constraints & Caching:**
Strictly adhering to the 1,000 API call limit, we strategically allocated resources to generate approx. 200 validation samples and 400 seed training samples. Furthermore, to prevent the loss of generated data due to potential runtime crashes (e.g., OOM), we implemented an **immediate caching system** that serializes and saves data to disk after each successful batch.

### **[참고] 보고서에 넣을 만한 핵심 키워드**
* **Transport Protocol:** gRPC blocking issue $\rightarrow$ Switched to REST.
* **Rate Limiting:** RPM (Requests Per Minute) bottleneck vs. RPD (Requests Per Day).
* **Throttling:** Using `time.sleep()` to respect API limits.
* **Robustness:** Error handling with Exponential Backoff.
* **Resource Allocation:** Strategic sampling under budget constraints (Validation vs. Seed).
* **Fault Tolerance:** Data serialization/caching to prevent API waste.
